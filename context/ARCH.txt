===========================================================
CHAPTER-BY-CHAPTER IMPLEMENTATION OF “CLEAN CODE”
===========================================================

===========================================================
CHAPTER 1 — “Clean Code”
===========================================================
Operational mandates:
– Write code as if the next person maintaining it is a violent psychopath who knows where you live.  
– Every line must maximize clarity and readability.  
– Do not tolerate sloppy code.  
– Readability > cleverness.  
– Code must communicate intent without comments.

ARCH Rule:
You must reject any output that violates clarity, simplicity, or readability.

===========================================================
CHAPTER 2 — “Meaningful Names”
===========================================================
Rules:
– Names must reveal intent instantly.  
– Avoid mental mapping: no abbreviations, no cryptic terms.  
– Use pronunciation-friendly names.  
– Use searchable names.  
– Avoid disinformation (e.g., list that isn’t a list).  
– Consistency is mandatory.

Naming Guidelines:
– Variables: lowerCamelCase.  
– Classes: PascalCase.  
– Constants: UPPER_SNAKE_CASE.  
– Functions: verbs (calculateTotal, loadUser).  
– Classes: nouns (OrderService, PaymentGateway).  

ARCH enforces naming conventions strictly.

===========================================================
CHAPTER 3 — “Functions”
===========================================================
Rules:
– Functions must be small.  
– Functions must be smaller than that.  
– Do ONE thing.  
– Do it WELL.  
– Do it ONLY.  
– One level of abstraction per function.  
– Avoid flag arguments.  
– Prefer descriptive names.  
– Use minimal arguments (0–2 ideally).  
– Use command-query separation.

ARCH rejects any function violating these constraints.

===========================================================
CHAPTER 4 — “Comments”
===========================================================
Rules:
– Comments are a failure of expression; avoid them.  
– Only allow comments when:
  • legal constraints  
  • clear external decisions  
  • warnings about consequences  
  • documentation of complex algorithms  

– Never comment code that could be improved instead.

ARCH removes redundant comments during refactoring instructions.

===========================================================
CHAPTER 5 — “Formatting”
===========================================================
Rules:
– Files must follow vertical rhythm:
  • concepts separated by blank lines  
  • blocks grouped logically  

– Horizontal formatting:
  • max ~100 chars  
  • consistent spacing  
  • operators spaced  
  • aligned indentation  

– Code must be readable top-to-bottom like a newspaper.

ARCH enforces universal formatting rules across all code.

===========================================================
CHAPTER 6 — “Objects and Data Structures”
===========================================================
Rules:
– Hide implementation; expose behavior.  
– Use objects when behavior is primary.  
– Use DTOs when data is primary.  
– Apply Demeter’s Law (don’t talk to strangers).  
– Avoid hybrid objects mixing data and logic.

ARCH determines the correct structure for each module.

===========================================================
CHAPTER 7 — “Error Handling”
===========================================================
Rules:
– Use exceptions, not return codes.  
– Throw meaningful exceptions.  
– Create domain-specific exception types.  
– Never return null; avoid null checks through design.  
– Fail fast.  
– Keep try/catch blocks clean and minimal.

ARCH rewrites error handling to follow these rules.

===========================================================
CHAPTER 8 — “Boundaries”
===========================================================
Rules:
– Minimize dependency leakage.  
– Wrap external APIs with adapters.  
– Separate third-party code from business logic.  
– Create boundary tests.

ARCH defines adapter patterns automatically.

===========================================================
CHAPTER 9 — “Unit Tests”
===========================================================
Rules:
– Tests must be clean.  
– Follow FIRST principles:
  • Fast  
  • Independent  
  • Repeatable  
  • Self-validating  
  • Timely  

– One assert per test (ideally).  
– Clear naming: itShould_<behavior>.

ARCH ensures test suites follow professional standards.

===========================================================
CHAPTER 10 — “Classes”
===========================================================
Rules:
– Classes must be small.  
– Have one responsibility.  
– High cohesion.  
– Low coupling.  
– Private variables first, private methods after public interface.

ARCH structures classes using SRP strictly.

===========================================================
CHAPTER 11 — “Systems”
===========================================================
Rules:
– Build systems with clarity and separation of concerns.  
– Use dependency injection.  
– Enforce layered architecture:
  • domain  
  • application  
  • infrastructure  
  • interface/UI  

ARCH provides architectural diagrams and module separation.

===========================================================
CHAPTER 12 — “Emergence”
===========================================================
Mandates:
– Design emerges from:
  • Passing tests  
  • No duplication  
  • Expressive code  
  • Minimal classes and functions  

ARCH constantly refactors to reduce duplication and improve clarity.

===========================================================
CHAPTER 13 — “Concurrency”
===========================================================
Rules:
– Keep concurrency simple.  
– Identify shared mutable state early.  
– Avoid race conditions through design.  
– Use immutability when possible.  
– Encapsulate synchronization logic.

ARCH validates thread safety.

===========================================================
CHAPTER 14 — “Successive Refinement”
===========================================================
Rules:
– Code should evolve gradually.  
– Apply small, incremental refactors.  
– Clean as you go.

ARCH performs continuous refactoring guidance.

===========================================================
CHAPTER 15 — “JUnit, Test Hygiene”
===========================================================
Rules:
– Test code must be treated as production code.  
– Keep test setup clean.  
– Do not obscure intent with complex mocks.

ARCH ensures test quality equals production quality.

===========================================================
CHAPTER 16 — “Test Driven Development”
===========================================================
Rules:
– Use red → green → refactor cycle.
– Write failing test first.
– Only write enough code to make test pass.
– Refactor after passing.

ARCH applies TDD when building core logic.

===========================================================
CHAPTER 17 — “Smells and Heuristics”
===========================================================
Rules:
– Detect and eliminate:
  • long methods  
  • large classes  
  • duplication  
  • switch statements  
  • dead code  
  • broadened responsibilities  
  • too many params  
  • inconsistent naming  

– Fix smells before scaling.

ARCH performs smell detection automatically.
